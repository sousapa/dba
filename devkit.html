Frequently asked questions

What is Base64 encoding and when should I use it?
Base64 converts binary or text data into a string using 64 printable ASCII characters. It's widely used to embed images in HTML or CSS, encode binary payloads in JSON APIs, transmit data in email attachments, and store credentials in HTTP Basic Auth headers. It's encoding — not encryption — so never use it to secure sensitive information.
What does a JSON validator actually check?
A JSON validator parses your text against the JSON specification (RFC 8259). Common errors it catches include: trailing commas after the last key-value pair, single quotes instead of double quotes around strings, unquoted keys, missing colons or commas, and unterminated strings. This tool also pretty-prints valid JSON and shows structural stats like depth and total key count.
What do the regex flags g, i, m, and s mean?
g (global): Find all matches, not just the first. i (insensitive): Case-insensitive matching, so /hello/i matches "Hello", "HELLO", etc. m (multiline): ^ and $ match start/end of each line rather than the whole string. s (dotAll): The . metacharacter also matches newline characters.
Is my data sent to a server?
The Base64 and Regex tools run entirely in your browser using JavaScript — your input never leaves your machine. The JSON validator sends your input to our server only to format it server-side, but no data is stored, logged, or shared. You can always validate locally using your browser's console: JSON.parse(yourString).
How do I decode a Base64 string in JavaScript?
In a browser: atob('SGVsbG8='). In Node.js: Buffer.from('SGVsbG8=', 'base64').toString('utf-8'). To encode: browser uses btoa('Hello') and Node.js uses Buffer.from('Hello').toString('base64').
What is a JWT and how does it work?
A JSON Web Token (JWT) is a compact, URL-safe token with three Base64url-encoded parts separated by dots: header.payload.signature. The header declares the signing algorithm (e.g. RS256). The payload carries claims — structured assertions about the user or session (who they are, what they can do, when the token expires). The signature is a cryptographic seal created with a secret or private key so that tampering can be detected. JWTs are stateless: the server validates the signature without a database lookup, which is why they are widely used in APIs and OAuth 2.0 / OpenID Connect flows.
Why does my JWT say "expired"?
The exp claim is a Unix timestamp (seconds since 1 Jan 1970 UTC). When the current time exceeds that value, the token is expired. Common causes: the token was designed to be short-lived (access tokens often last 15–60 minutes), your system clock is wrong, or you are reusing a token from a previous session. Request a new token from your auth server — expired tokens cannot be extended without re-authentication or a refresh token flow.
Why is the "none" algorithm dangerous?
When alg is set to none in the header, the JWT has no signature. A malicious actor can craft a token with any claims they like (e.g. elevating their role to admin), set alg: "none", and submit it. If the server accepts unsigned tokens, it will trust the forged claims. This attack was used to break several real-world systems. Secure JWT libraries reject alg: none by default — ensure yours does too and never whitelist it.
What is the difference between HS256 and RS256?
HS256 (HMAC-SHA256) is symmetric: the same secret is used to both sign and verify tokens. Any service that can verify must also be trusted with the secret — meaning it could forge tokens. RS256 (RSA-SHA256) is asymmetric: a private key signs, a public key verifies. Services only need the public key to verify — they cannot forge. For single-service or internal systems HS256 is fine with a strong random secret. For public APIs, multi-service architectures, or when third parties verify your tokens, prefer RS256 or ES256.
